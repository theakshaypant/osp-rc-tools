# osp-release-audit

Toolkit for auditing and managing OpenShift Pipelines releases. Combines Go CLI tools for commit/Jira analysis with Claude Code skills that automate the full release verification workflow — from hack repo config through stage/production releases and QA handover.

## How it works

Each OpenShift Pipelines release (e.g., 1.22.4) spans ~20 component repos — Tekton Pipeline, Triggers, Chains, operator, etc. The release configuration lives in [`openshift-pipelines/hack`](https://github.com/openshift-pipelines/hack) under `config/downstream/releases/{major.minor}.yaml`, which maps each component to its upstream branch and downstream productization repo.

The tool uses two strategies to find what commits belong to a release:

1. **Tag-based (preferred):** Looks for a tag `v{version}` on the downstream repo and reads the `head` file at that tag to get the upstream commit SHA. This is deterministic and fast.
2. **Time-based (fallback):** When tags don't exist, derives date bounds from the hack repo's commit history (when `release-tag` was bumped between versions) and finds the most recent `head` file commit before that date.

Once it has the commit range for each component, it resolves PRs (including original PRs for cherry-picks), links commits to Jira tickets via the PR custom field, and fetches release note fields from Jira.

### Tag-based resolution

The tool works best when downstream repos have release tags (e.g., `v1.22.4`). With tags, the commit boundaries are exact. Without them, the tool falls back to time-based resolution using hack repo history, which can be imprecise if the timing of `release-tag` bumps doesn't perfectly bracket the actual release window.

## Prerequisites

- **Go 1.22+**
- **GitHub CLI (`gh`)** — authenticated via `gh auth login`
- **Jira API token** (optional but recommended) — for ticket linking and release notes

## Setup

```bash
git clone https://github.com/theakshaypant/osp-rc-tools.git
cd osp-rc-tools
go build ./...
```

Create a `.env` file with your credentials:

```bash
# GitHub (optional — falls back to `gh auth token`)
GITHUB_TOKEN=ghp_...

# Jira (optional — tool works without it but skips ticket linking)
JIRA_URL=https://redhat.atlassian.net
JIRA_EMAIL=you@redhat.com
JIRA_TOKEN=ATATT...

# Jira custom field IDs (defaults shown — override if your instance differs)
JIRA_PR_FIELD=customfield_10875
JIRA_RN_TEXT_FIELD=customfield_10783
JIRA_RN_TYPE_FIELD=customfield_10785
JIRA_RN_STATUS_FIELD=customfield_10807

# Konflux cluster (for one-click-release snapshot verification)
KONFLUX_SERVER=https://api-toolchain-host-operator.apps.kflux-prd-rh02.0fk9.p1.openshiftapps.com
KONFLUX_TOKEN=sha256~...    # get token: https://oauth-openshift.apps.kflux-prd-rh02.0fk9.p1.openshiftapps.com/oauth/token/display

# GitLab (for RPA/Pyxis verification — read-only)
GITLAB_URL=https://gitlab.cee.redhat.com
GITLAB_TOKEN=glpat-...
```

Source it before running:

```bash
source .env
```

## Usage

### Audit a patch release

```bash
go run cmd/audit/main.go 1.22.4
```

Compares commits between 1.22.3 and 1.22.4 for every component. Output is written incrementally to `reports/release_1.22.4.json`.

### Audit a minor release

```bash
go run cmd/audit/main.go 1.23.0
```

When the patch number is `0`, the tool treats it as a minor release audit. It compares the full commit range from the end of the previous minor (1.22.x) through 1.23.0, using each version's respective upstream branches.

### Re-fetch Jira data on an existing audit

```bash
go run cmd/jira-update/main.go reports/release_1.22.4.json
```

Re-runs only the Jira steps (release note fields + fixVersion ticket search) on an existing audit JSON. Useful when Jira fields were updated after the initial audit.

## Output

### JSON (`reports/release_{version}.json`)

The primary output. Contains per-component commit lists with:

- Commit metadata (SHA, message, author, date)
- PR links (including original upstream PR for cherry-picks)
- Jira ticket links (resolved via the PR custom field)
- Release note fields from Jira (text, type, status)
- Unsynced commits (upstream commits not yet in downstream)
- Unmatched Jira tickets (fixVersion tickets with no linked commit)

### Markdown report (`reports/audit-release-{version}.md`)

Generated by the `audit-release` Claude Code skill (see below). A human-readable report with per-component tables showing commits, Jira links with confidence annotations, and release notes with source tracking.

## What the tool resolves

For each component in a release:

| Step | What | How |
|------|------|-----|
| Commit range | Upstream commits in this release | Tag on downstream repo, or time-bounded `head` file history |
| PRs | Pull request that merged each commit | GitHub `ListPullRequestsWithCommit` API |
| Original PRs | Upstream PR for cherry-picked commits | Cherry-pick trailer SHA, or PR body/title references |
| Jira tickets | Linked Jira issue | Jira PR custom field search, fixVersion query |
| Release notes | RN text, type, status | Jira custom fields on the linked ticket |

## Claude Code skill: `audit-release`

The Go tool produces structured JSON. The `audit-release` skill picks up where it leaves off, enriching the data with additional Jira searches and release note generation.

```
/audit-release @reports/release_1.22.4.json
```

What the skill adds:

1. **Tier 2 Jira search** — Extracts Jira keys from PR titles and bodies via `gh pr view`
2. **Tier 3 Jira search** — Keyword-based JQL queries against Jira for still-unlinked commits
3. **Release note extraction** — Pulls release notes from PR description sections (````release-note` blocks, `## Release Notes` headings)
4. **Release note generation** — AI-generates 1-3 sentence user-facing release notes when none are found
5. **Type classification** — Assigns release note type (Bug Fix, CVE, Feature, Enhancement, etc.) from PR labels and content
6. **Markdown report** — Generates a per-component audit report with all findings annotated by source

Each component is processed by an isolated subagent to keep context manageable for large releases. Results are written back to the JSON incrementally with provenance fields (`jira_source`, `rn_source`) so the report generator can annotate confidence without re-deriving it.

## Claude Code skills

Beyond the Go CLI, most of the release workflow is driven by Claude Code skills invoked via slash commands:

### Release checklist (`/release-checklist {version}`)

A 16-step sequential release verification workflow. Queries GitHub, GitLab, Konflux cluster, and Jira to validate each step — from hack repo config through production release and advisory. Stops at the first step that needs action and generates a report at `reports/checklist-{version}.md`.

Can also be run by individual step group:

| Command | Steps | What it checks |
|---------|-------|----------------|
| `/release-checklist:check-hack` | 1-2 | Hack repo release config |
| `/release-checklist:check-konflux-config` | 3-5 | Konflux cluster config, RPA, Pyxis |
| `/release-checklist:check-components` | 6-8 | Component PRs, OLM bundle version, upstream sync, nudges |
| `/release-checklist:check-builds` | 9-10 | Operator version, build SHA validation |
| `/release-checklist:check-olm` | 11-12 | OLM catalog render, index images, code freeze |
| `/release-checklist:check-releases` | 13-16 | Stage/prod releases, QA handover, advisory |

### One-click release (`/one-click-release {version}`)

Walks through the release workflow in stages with both **verify** (read-only, automatic) and **execute** (write, requires approval) commands. Currently implements Config (steps 1.1–1.13), Build (steps 2.1–2.7), and Image Copy (steps 3.1–3.2) stages — from hack repo config through CLI version bumps, snapshot verification, CDN release, code freeze, and index image copy to quay.io. See [docs/skills/one-click-release.md](docs/skills/one-click-release.md) for full details.

### Other skills

| Command | Description |
|---------|-------------|
| `/audit-release` | Enrich audit JSON with Jira links, release notes, and markdown report |
| `/build-diff` | Compare two Konflux snapshots side by side |
| `/new-build` | Verify build health and generate diff for QE |
| `/jira-gaps` | Identify Jiras missing fixVersion or release note fields |
| `/cli-release` | Track CLI release progress (serve-tkn-cli binaries to CDN) |
